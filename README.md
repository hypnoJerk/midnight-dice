# Midnight: 1-4-24 Multiplayer Retro PWA

**Midnight** is a premium, real-time multiplayer, mobile-first Progressive Web App (PWA) built to play the authoritative **1-4-24 (Midnight)** dice game. It integrates an immersive **3D React Three Fiber (R3F) physics dice canvas** with a **retro CRT phosphor video game aesthetic** (CRT scanlines, flickering overlays, and responsive amber/green phosphor styles). 

Authentication is authoritatively governed by a secure, password-protected session layer, saving all statistics and scores to a persistent PostgreSQL high-score leaderboard.

---

## 🎮 Game Rules & Engine

Midnight is played with six six-sided dice, coordinated entirely by the server's authoritative state machine to prevent client-side manipulation.

1. **Standard Turn**:
   * One player rolls at a time. The first roll throws all six dice.
   * **Mandatory Action**: After each roll, the active player must keep/set aside at least one die. Unkept dice are re-rolled.
   * **Qualification Phase**: To qualify for scoring, a player must set aside both a **1** and a **4** before the turn ends.
   * **Disqualification (Bust)**: If a player fails to keep both a **1** and a **4** once all six dice are set aside, they are disqualified (`BUST` with `0` points) for that round.
   * **Score Calculation**: If qualified, the remaining four dice are totaled to determine the score (Maximum score is **24**).
   * **Win Condition**: The qualified player with the highest four-dice total wins.

2. **The Shootout (Tie-Breaker)**:
   * If two or more players tie at the highest score (or if everyone is disqualified at 0 points), they enter a **Shootout**.
   * Each shootout participant rolls all six dice in one go.
   * **Qualification rules are suspended**; the shootout score is the simple sum of all six dice (maximum 36).
   * Shootouts continue in a loop until the tie is broken.

## 🛠️ Architecture & Monorepo Directory Layout

Midnight is designed as a decoupled monorepo managed via **npm workspaces**:

```
project_code/
├── client/          # React Frontend (Vite, TS, R3F, Cannon Physics, Web Audio API, PWA manifest)
├── server/          # Node.js Backend (Express, TS, Socket.io, Drizzle ORM, PBKDF2 cryptography)
├── shared/          # Shared Typings (Unified network payloads, state machine interfaces, and types)
├── nginx/           # Reverse Proxy default routing mapping (proxies ports 5173, 3001, and web sockets)
└── docker-compose   # Container Orchestration mapping (Local multi-container dev/prod environment)
```

### Technical Highlights
* **3D Cannon Physics lerping**: Powered by `@react-three/fiber` and `@react-three/cannon`. The physical dice are simulated dynamically. Once they settle, the client smoothly interpolates (lerps) the rotation to display the exact authoritative face value generated by the server.
* **Spectator Battery Saver**: The 3D physics canvas mounts *only* for the active player. Spectators receive a synchronized static scoreboard, saving CPU and battery.
* **8-Bit Procedural Web Audio**: Generates arcade-style click beeps, roll clatters, Major arpeggio success chimes, and dq sawtooth buzzers procedurally via the **Web Audio API** (zero assets, 100% offline).
* **Zero-Dependency Password Hashing**: Utilizes Node's native `crypto` module with standard PBKDF2 (SHA-512) password hashing, eliminating the risks of native compilation failures associated with packages like `bcrypt`.

## 🚀 Quick Start (Local Development)

The entire environment is containerized and orchestrated. Follow these steps to launch the monorepo locally:

### 1. Prerequisites
Ensure you have **Docker** and **Docker Compose** installed.

### 2. Start the Stack
Navigate to the `project_code/` directory in your terminal and run:
```bash
docker compose up --build
```
This automatically initializes:
1. **PostgreSQL** instance on port `5433`.
2. **Express Server** on port `3001` (running safe startup schema upgrades).
3. **React Client** (Vite Dev Environment) on port `5173`.
4. **Nginx Reverse Proxy** on port `8080` to route traffic.

### 3. Open and Play
Open your browser and navigate to:
```
http://localhost:8080
```
Enter an agent codename and password to create an account, then host or join a room code!

## 🌐 Production Deployment (DigitalOcean Standalone Droplet)

For a single standalone server instance on a DigitalOcean Droplet:

### 1. Provision the Droplet
1. Create a new Droplet on DigitalOcean (Ubuntu 22.04 LTS, Basic shared CPU, 1GB RAM is sufficient).
2. Enable IPv4 and configure SSH access.

### 2. Install Docker & Compose
SSH into your droplet and install Docker:
```bash
sudo apt update
sudo apt install -y docker.io docker-compose-v2
sudo systemctl enable --now docker
```

### 3. Clone and Run
Clone your repository to the droplet, navigate to `project_code/`, create a `.env` file for the backend settings, and start the production build:
```bash
docker compose -f docker-compose.yml up -d --build
```
Your game will now be live on Port `8080` (or `80` if you map Nginx ports in `docker-compose.yml` accordingly).

## 🛰️ Advanced Shared Infrastructure Setup (Multi-Client DigitalOcean Server)

In production setups, it is common to host multiple applications on a single DigitalOcean Droplet under a **shared-infrastructure topology**:
* **One Global Nginx Container**: Acts as the reverse proxy router for all subdomains and SSL certificates.
* **One Global PostgreSQL Container**: Hosts databases for all separate websites under isolated tablespaces.
* **Isolated Client/Server Containers**: Running independently, connected only to the shared services via bridge networks.

### Topology Architecture

```
Client Browser
     │ (Port 80/443 - e.g. midnight.yourdomain.com)
     ▼
┌───────────────────────────────────────────────┐
│       Shared Nginx Reverse Proxy Router       │
└──────────────────────┬────────────────────────┘
                       │ (Proxy Pass via Docker Net)
        ┌──────────────┴──────────────┐
        ▼                             ▼
┌──────────────┐              ┌──────────────┐
│  Midnight    │              │  Midnight    │
│  Client      │              │  Server      │
│  Container   │              │  Container   │
└──────────────┘              └──────┬───────┘
                                     │ (Database Query)
                                     ▼
                      ┌─────────────────────────┐
                      │  Shared Postgres Db     │
                      │  ┌───────────────────┐  │
                      │  │   midnight_prod   │  │
                      │  └───────────────────┘  │
                      └─────────────────────────┘
```

Follow this step-by-step guide to adapt Midnight to this shared infrastructure:

### 1. Segregate the Shared PostgreSQL Database
Rather than spinning up another PostgreSQL service inside Midnight's docker-compose, create an isolated database and database user inside your existing global PostgreSQL container.

Connect to your shared Postgres instance as superuser and execute:
```sql
CREATE DATABASE midnight_prod;
CREATE USER midnight_user WITH ENCRYPTED PASSWORD 'your_secure_password';
GRANT ALL PRIVILEGES ON DATABASE midnight_prod TO midnight_user;
```

### 2. Configure Docker Networks
To allow Midnight's backend container to talk directly to your global PostgreSQL container and allow your global Nginx router to proxy traffic to Midnight's containers, they must share Docker bridge networks.

Assuming your global PostgreSQL container runs on a network named `shared-db-network`, and your global Nginx container runs on a network named `shared-nginx-network`:

Verify or create the external networks on your Droplet:
```bash
docker network create shared-db-network
docker network create shared-nginx-network
```

### 3. Adapt Midnight's `docker-compose.yml` for Shared Resources
Remove the local `db` and local `nginx` services from Midnight's `docker-compose.yml` file. Instead, configure it to hook into the external networks.

Rewrite the `docker-compose.yml` as follows:
```yaml
version: '3.8'

services:
  server:
    build:
      context: ./server
      dockerfile: Dockerfile
    container_name: midnight_server
    restart: always
    environment:
      NODE_ENV: production
      # Connect directly to the shared postgres host over the shared network
      DATABASE_URL: postgresql://midnight_user:your_secure_password@shared_postgres_container_name:5432/midnight_prod
      PORT: 3001
    networks:
      - shared-db-network
      - shared-nginx-network
    expose:
      - "3001"

  client:
    build:
      context: ./client
      dockerfile: Dockerfile
    container_name: midnight_client
    restart: always
    environment:
      VITE_API_URL: https://midnight.yourdomain.com/api
    networks:
      - shared-nginx-network
    expose:
      - "5173"

networks:
  shared-db-network:
    external: true
    name: shared-db-network
  shared-nginx-network:
    external: true
    name: shared-nginx-network
```
*(Replace `shared_postgres_container_name` with the actual `container_name` of your global PostgreSQL container).*

### 4. Configure the Shared Nginx Server Block
Add a new virtual host configuration inside your global Nginx proxy container (e.g. at `/etc/nginx/conf.d/midnight.conf`) to map your subdomain to the client and server:

```nginx
server {
    listen 80;
    server_name midnight.yourdomain.com;

    # Redirect all HTTP traffic to HTTPS (Highly Recommended)
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name midnight.yourdomain.com;

    # Add your Let's Encrypt / SSL Cert configurations here
    ssl_certificate /etc/letsencrypt/live/midnight.yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/midnight.yourdomain.com/privkey.pem;

    # 1. Route Frontend PWA Requests
    location / {
        proxy_pass http://midnight_client:5173;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }

    # 2. Route REST API Requests
    location /api {
        proxy_pass http://midnight_server:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 3. Route Real-time WebSocket Traffic
    location /socket.io/ {
        proxy_pass http://midnight_server:3001/socket.io/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Reload the shared Nginx proxy:
```bash
docker exec shared_nginx_container_name nginx -s reload
```

## 🗄️ Database Management & Schema Resets

Midnight utilizes **Drizzle ORM** internally. However, table maintenance and upgrades are fully managed **programmatically** at server bootstrap for maximum deployment convenience.

### Schema Auto-Migrations
When the `midnight_server` container boots, it automatically connects to Postgres, inspects the existing tables (`users`, `matches`, `match_participants`), and runs safe startup migrations (e.g. adding columns, unique index constraints, or primary key defaults) without breaking existing data structures.

### Wiping and Resetting the Database
If you ever need to purge the entire database and start over from a pristine clean slate:

1. **Via Command Line**: Set the `PURGE_DB=true` or `RESET_DB=true` environment variables when running the server:
   ```bash
   PURGE_DB=true node dist/index.js
   ```
2. **Via Docker Compose**: Add the environment flag temporarily to your `docker-compose.yml` under the `server` environment block:
   ```yaml
   environment:
     - PURGE_DB=true
   ```
   Spin up the container once (`docker compose up server`) to trigger the purge, then remove the environment variable to maintain persistence going forward!

## 🧪 Testing Pipeline

We maintain a strict testing pipeline using **Vitest** to assert scoring, WebSockets, and database transactions:

### Run Tests Locally
1. Install monorepo root dependencies:
   ```bash
   npm install
   ```
2. Execute the Vitest suite against the backend package:
   ```bash
   npm run test -w server
   ```

Our suite of **20 tests** covers:
* **Scoring Rules**: Validating 24-point maxima, missing `1` or `4` DQ rules, shootout sums.
* **Authentication Flow**: Asserting registration, login, PBKDF2 hash verification, and duplicates rejection.
* **Socket.io Handshakes**: Asserting player rolls authorization, index bounds validations, turn-keeping states.
* **Database Queries**: Verifying transaction-safe Drizzle inserts for matches, participants, and profile increments under game-over hooks.
